# **Integrating NodeTool, Flowise, n8n, and Node-RED into a VS Code Extension**

## **Overview**

We compare four open-source, Node.js-based visual workflow/canvas tools – NodeTool, Flowise, n8n, and Node-RED – in terms of how easily each can be embedded into a VS Code OSS extension. The goal is to allow users to interact with the full visual canvas UI and execute workflows entirely inside VS Code (e.g. via a WebView), without requiring any external cloud services or remote servers (all execution is local). Key factors in this evaluation include:

* WebView UI Embeddability: Can the tool’s web-based UI run inside a VS Code WebView (e.g. as a local web app or via an embedded iframe)?  
* Execution Engine Integration: Can the workflow execution engine run within the extension’s context (as a Node.js module, child process, etc.)?  
* Offline/Local Execution: Ability to execute workflows locally without needing cloud services or external backend.  
* Persistence & File Management: How workflows are stored (files vs. database) and whether they can be saved in the workspace for version control.  
* Limitations & Integration Challenges: Potential blockers like authentication UIs, fixed ports, heavy dependencies (e.g. requiring Python or Electron), or licensing constraints.  
* Prior Art: Any known examples or community attempts to integrate the tool with VS Code.

Below, we examine each tool on these points, followed by a comparison table and recommendations.

## **NodeTool (Visual AI Workflow Builder)**

NodeTool is an open-source visual builder for AI pipelines (AGPL-3.0 licensed). It lets users drag-and-drop nodes for AI tasks (image generation, text processing, etc.) and emphasizes running those workflows locally or on self-hosted infrastructure.

* UI Embeddability: NodeTool provides a rich web-based UI (it even has an Electron app for desktop). In principle, this UI can be loaded in a VS Code WebView since it’s just HTML/JS. However, the NodeTool UI is not purely static – it requires a backend server. NodeTool’s “Editor” UI communicates with a local API server started by the nodetool serve command. To embed the UI, an extension would likely launch NodeTool’s backend server (which runs on a local port, default around 7777 by configuration) and then load the front-end. The Electron mode confirms the UI can run in an embedded browser context. The main challenge is that the WebView must either point to the NodeTool local server (e.g. http://localhost:7777), or serve the UI files itself. Using an iframe or direct WebView navigation to the local URL is possible but may require loosening VS Code’s content security policy to allow the connection.  
* Execution Engine Integration: NodeTool’s execution engine is somewhat hybrid – much of the workflow logic (especially AI model execution) runs in Python processes. The NodeTool project includes a Python backend (nodetool-core and related packages) that actually executes nodes and workflows. The VS Code extension would need to ensure this backend is available. One approach is bundling NodeTool’s Python environment or instructing users to install it, then launching nodetool serve as a subprocess. The engine runs locally on the user’s machine (NodeTool is “private by default” – models run on your hardware unless configured otherwise). Integrating the engine is not as simple as a single Node.js require(), due to the Python dependency. That said, once nodetool serve is running, the extension could use NodeTool’s API (REST/WebSocket) to execute workflows or even drive them programmatically. No external cloud backend is needed – NodeTool is designed to work offline (it can even run heavy models locally given the right hardware).  
* Local-Only Operation: NodeTool fully supports local execution. Users can run models on local CPU/GPU or choose cloud APIs per node. By default it doesn’t require any cloud connection. The VS Code integration can leverage this by running all tasks on the user’s machine. (If a user opts to use a cloud API node, that’s up to them – but the platform itself doesn’t force any cloud service.)  
* Persistence & Workspace Integration: NodeTool stores workflows internally as JSON (every workflow is represented in JSON). It has a “Workflows” panel in the UI for saved workflows. Under the hood, these might be stored in a local database or files by the NodeTool server (the documentation hints at a storage system, possibly a local folder or SQLite DB for assets and flows). There isn’t a one-click way to have NodeTool save each workflow as a .json file in the VS Code workspace – you’d likely need to use NodeTool’s export/import functionality to persist flows to the project. An advanced integration could call NodeTool’s API to load/save workflow JSON to files (or perhaps configure NodeTool’s storage directory to point to the workspace). But out-of-the-box, NodeTool doesn’t use the VS Code workspace; it manages workflows in its own environment, so an extension would have to bridge that gap if file-based version control is required.  
* Integration Challenges: 1\) Dependency Footprint: NodeTool’s reliance on a Python (and Conda) environment is a major integration hurdle. Unlike the other tools (which are pure Node.js), NodeTool would force the extension either to bundle a Python runtime or instruct users through a potentially heavy setup. This could complicate installation for end-users of the extension. 2\) WebView Communication: If the WebView loads the NodeTool UI from localhost:7777, we must ensure VS Code allows that. The extension might use an iframe in the WebView pointing to the NodeTool URL – similar patterns have been used (the Node-RED extension, for example, iframes a local server). Ensuring that the NodeTool UI’s scripts can connect back to the local API (which would be same-origin if loaded via the local server) is straightforward, but if the extension attempted to load UI assets directly, it would need to handle CORS. 3\) Authentication: NodeTool’s local Editor API can run with no auth token when configured in “trusted local” mode, so probably you wouldn’t need to deal with login prompts. This is good, as any auth UI in the WebView (like a login screen) would make integration harder. 4\) Licensing: NodeTool is AGPL v3.0. This strong copyleft license means that if you distribute NodeTool as part of your extension, your extension likely also needs to be AGPL-compliant. For a VS Code OSS extension (which can be open source, but often under MIT or similar), this could be a blocker unless you’re willing to adopt AGPL or structure the integration to avoid derivative works issues. In short, legally it’s more restrictive than Apache or MIT. 5\) Community Support: NodeTool is relatively new (fewer contributors and users so far). There are currently no known VS Code extensions or community examples to follow, so you’d be breaking new ground. This means more initial effort compared to tools that others have already embedded.

Summary for NodeTool: Technically feasible (the UI is web-tech and the engine can run locally), but integration would be moderately difficult due to the need to manage a Python backend inside an extension and licensing constraints. If your focus is on AI workflows that require on-device ML (images, video, etc.), NodeTool offers capabilities there that others don’t, but expect extra work to embed it seamlessly.

## **Flowise (LangChain/LLM Workflow Builder)**

Flowise is an open-source visual builder for LLM-based applications and agents. It provides a drag-and-drop interface to chain together prompts, models, document loaders, etc., targeting use cases like chatbots and AI assistants. Flowise’s codebase is split into a Node.js backend (Express server) and a React frontend. It is licensed under Apache 2.0, which is very friendly for integration (no copyleft restrictions).

* UI Embeddability: The Flowise UI runs as a web app. When you start Flowise (via npx flowise start or similar), it launches a local web server and serves the React app on a port (default 3000 by HTTP). In VS Code, the simplest approach would be to run the Flowise server in the background and load its UI in a WebView. For example, an extension could spawn the Flowise process and then use an iframe or direct webview content pointing to http://localhost:3000. The Flowise docs note that the allowed origins for iframes can be configured via an environment variable (IFRAME\_ORIGINS) – this suggests embedding is a considered use case. We might need to set IFRAME\_ORIGINS=\* (or a specific VS Code webview origin scheme) to allow the WebView to display it. Alternatively, since we have access to the source, one could theoretically build the Flowise React app as static files and serve them via the extension’s WebView directly. However, that would require also re-implementing the backend API calls (or forwarding them to a local server anyway). The more straightforward path is the one proven by Node-RED’s integration: run Flowise’s server on a loopback interface and point a WebView to it. One must ensure the WebView’s content security policy allows connecting to http://localhost:PORT. (VS Code typically disallows arbitrary http, but the extension can specify allowed domains or use the WebView IPC if needed.) In summary, embedding Flowise’s UI is very possible, but some configuration is needed to avoid CSP and CORS issues. Notably, Flowise also provides an embeddable chat widget for websites, but that is for end-user chat interfaces – not the builder UI. For our purposes, we want the full builder. No built-in VS Code extension exists yet, but the UI tech stack is standard (React+REST), so it’s on par with a typical web app integration.  
* Execution Engine Integration: Flowise’s backend is a Node.js process that orchestrates the workflows. This can be started within the extension context. Because Flowise is distributed as an npm package/CLI (flowise), one approach is to include it as a dependency and invoke its start programmatically (or simply spawn flowise start). Under the hood, this will initialize an Express server hosting both the API and the static UI. The extension can manage this process lifecycle (start on extension activation, shut down on extension deactivation). Importantly, Flowise does not need any external services – it can load and run the flows itself. For example, Flowise supports various “components” (nodes) including calls to OpenAI, local vector databases, etc., all handled by the local server. The engine uses LangChainJS or similar libraries to execute LLM calls. There’s evidence of integration with VS Code in pieces: one extension (LLMCoderAgent) expects a Flowise server to be running and then interacts with it via HTTP API. In that case, the Flowise engine was outside VS Code, but it shows that Flowise exposes a local API which an extension can leverage. For full integration, we’d run that API internally. From a Node perspective, this is easier than NodeTool’s case (no extra Python to manage). It’s just a Node process or possibly even a set of Node modules. (Flowise’s code is monorepo; we might be able to import the server module and mount it on an Express instance inside the extension, customizing the port, etc., instead of launching a separate process.)  
* Local Execution: Flowise is designed to be self-hosted. Users can run it locally and connect it to local or cloud resources as they choose. By default it uses no cloud services unless your workflow includes nodes that call out (like an OpenAI API node). It supports local database and even LocalAI for running open-source models on your machine. So an offline-capable scenario is definitely supported – e.g., you could use Flowise with a local LLM endpoint and local data, all within VS Code offline. The extension just needs to ensure the Flowise server has the right environment (API keys or paths for any local model backends the user wants to use, but those are user configurations, not required infrastructure).  
* Persistence & File Structure: Flowise by default uses a SQLite database to store all its data (flows, chat histories, etc.). The default location is in the user’s home directory (\~/.flowise on a typical system). Each new workflow built in the UI is saved into that DB. This means without modification, the workflows aren’t represented as separate files in the workspace. However, Flowise does support switching the database – environment variables allow using MySQL/Postgres or changing the SQLite file path. A VS Code extension could leverage this by pointing Flowise’s DATABASE\_PATH to somewhere in the workspace (or in the extension storage directory) so that workflow definitions are saved in a known location. Even if they remain in a single SQLite file, that file could be in the workspace for easy backup or versioning. Additionally, Flowise flows can be exported via the UI or API (they are essentially JSON structures that correspond to a chain of nodes). In practice, you could have an export/import feature in the extension to dump a flow to a JSON file in the workspace (and even edit it, though editing might be complex). There is community interest in generating Flowise flows programmatically, which implies the JSON schema of flows is known. So, while not natively file-based, it’s feasible to integrate with VS Code’s file model via export or DB configuration. One more note: Flowise doesn’t have a “projects” concept as Node-RED does; it’s more like a single-user app instance with multiple flows stored in one place. So if a user had multiple VS Code projects, each wanting its own flows, the extension might need to reinitialize or use different DB paths per project.  
* Known Limitations & Potential Issues: 1\) Port and Connectivity: By default Flowise binds to port 3000, which could conflict if something else is running. The extension should choose a port (Flowise allows PORT to be set via env), possibly picking an available port dynamically. The WebView must then load that exact localhost:port URL. This is doable, just needs coordination. 2\) CSP/CORS: As mentioned, the extension HTML can include an iframe or redirect to the Flowise URL. We’ll need to allow http://127.0.0.1:PORT in the WebView’s content security policy. Also, Flowise’s server might need to allow the WebView origin. If using an iframe, the WebView origin is something like vscode-webview://.... Setting IFRAME\_ORIGINS="\*" or a specific scheme should handle it. If the WebView directly connects (without an iframe), e.g. by having \<meta http-equiv="Content-Security-Policy" ...\> to allow the script sources from localhost, that could also work. These are solvable with configuration. 3\) Authentication/UI: By default, Flowise does not require user authentication for its local instance – it’s open on localhost:3000 without login. (Flowise Cloud has auth, but that’s not relevant here.) This simplifies integration – we won’t be prompted to log in within the WebView. 4\) Multi-user or security considerations: Because Flowise is open on a port, one should ensure it’s only bound to localhost (which it is, by default). If the extension left it running, it might be accessible outside VS Code on that port. But since it’s local, this is usually fine (and you could shut it down when the extension deactivates). 5\) No existing VS Code extension: As of now, there isn’t a known example of the full Flowise UI in VS Code. The closest is the *LLM Coder Agent* extension, which only uses Flowise’s API for chat, not the UI. So, like NodeTool, a full integration would be pioneering territory – but given Flowise’s simpler tech stack and permissive license, it’s likely easier than NodeTool. 6\) Memory/Performance: Running Flowise inside VS Code will consume additional memory (it’s essentially like running a small webserver \+ React app). VS Code can handle this, similar to running a Live Server or the Node-RED integration, but it’s worth noting for very large workflows or heavy LLM usage you might hit resource limits. In testing, Flowise is reasonably lightweight for what it does.

Summary for Flowise: Flowise should be relatively straightforward to integrate. You have to run a local Node process and embed a web UI – patterns that have precedent. There are no major architectural blockers. With Apache 2.0 licensing, it’s legally simple to include. The main tasks are handling the WebView \<-\> Flowise server interaction and deciding how to store workflows. If your extension is focused on AI workflows (LLM agents, chatbots) and you want a modern UI with minimal fuss, Flowise is a strong candidate.

## **n8n (Workflow Automation Platform)**

n8n is a popular general-purpose workflow automation tool (often compared to Zapier or Node-RED but with a broader node library for SaaS integrations). It is source-available but not fully open source; it uses the “Sustainable Use License” (a fair-code license). This license permits free personal and small-scale use but restricts large-scale commercial use and isn’t OSI-approved. That’s a first consideration for integration: including n8n in your extension might impose licensing constraints (you couldn’t simply MIT-license your extension if it bundles n8n code, since n8n’s license would apply to its parts). Assuming license issues can be navigated or are acceptable, here’s how n8n stands on integration:

* UI Embeddability: n8n’s UI is a web application (built likely with Vue.js) that runs in tandem with its Node.js backend. Normally, one accesses it via browser at http://localhost:5678 (default) when the n8n server is running. Unlike Node-RED or Flowise, embedding n8n’s full editor is known to be tricky. The n8n team/community have explicitly noted that *you cannot safely embed the full n8n editor in another app* without considerable effort, because it requires a backend and handles authentication tokens. Starting with n8n v1.0+, when you run n8n locally you are prompted to create an owner account and log in via the web UI. The editor will not function until you authenticate, as a security measure (to protect potentially sensitive automation credentials). This means that if we tried to load the n8n UI in a VS Code WebView, we’d get a login screen. There is no official way to disable authentication entirely in n8n (the team discourages it for security). Workarounds might include auto-generating an access token or using a dummy login, but these add complexity (you might need to intercept the login page in the WebView or pre-populate credentials). Technically, one could imagine launching n8n with an environment variable that sets up a default user and auto-logs in via a query param, but n8n doesn’t provide a documented method for that in local deployments. Another approach is to use n8n’s Public API (if available) to bypass the UI entirely, but then you lose the visual canvas. In short, while the VS Code WebView could display the n8n editor (there is nothing preventing an iframe to http://localhost:5678 from a technical standpoint), the requirement of logging in (and managing sessions/cookies inside the WebView) is a significant hurdle. Moreover, even aside from auth, the n8n UI is not designed for embedding; it might assume a full browser environment. For example, performing OAuth2 for certain nodes might try to open new windows. These flows could break or be awkward in the confined WebView. Real-world note: No known VS Code extension embeds n8n’s own UI at this time. The community question *“How to embed the workflow UI in my application?”* concluded that it’s not straightforward and instead pointed to a limited embed demo component for read-only workflows (which isn’t useful for editing/running flows fully).  
* Execution Engine Integration: On the positive side, n8n’s core engine is just a Node.js application, so it can be run locally from an extension. In fact, n8n has a CLI and even a desktop app (which is essentially an Electron wrapper around the local web server). So spinning up n8n in the background is feasible: the extension could bundle n8n or require it as a dependency, then spawn n8n start \--tunnel=false (to ensure it doesn’t try to use their cloud tunneling) with appropriate environment settings (maybe specifying SQLite DB path, etc.). The workflows would then execute in that process. Communication with the running n8n could be through its REST API. For example, the extension could create new workflows by calling the API, or listen for events. There’s also the concept of triggering executions via CLI or SDK, but n8n is primarily designed to run as a service. If embedding the full UI is too hard, an alternative is to control n8n via its API and present a different UI in VS Code. One existing extension (n8n-as-code) takes this approach: rather than show the n8n editor, it lets you edit workflow JSON in VS Code and then syncs with a live n8n instance via API. It even provides a basic visual preview of the workflow graph in a WebView for validation (not the interactive editor, but a real-time rendering of the flow). This confirms that the engine can run separately and be manipulated from VS Code. The drawback is you lose the nice drag-and-drop interface that is a big selling point of n8n.  
* Local Execution: n8n can run fully offline and local. By default it will use a SQLite database if no external DB is configured, so it doesn’t need any external service. All nodes that interact with third-party services (like a Google Sheets node, etc.) will naturally require internet or credentials, but that’s up to the user’s workflow. The platform itself works offline (for example, you can use n8n to manipulate local files, run code, etc., with no cloud). So meeting the “no cloud required” condition is not a problem for n8n. You just run it as a local service.  
* Persistence & Workspace: n8n stores workflows in its database. If using SQLite, that’s typically a file named database.sqlite in the n8n user directory (like \~/.n8n/ by default). The extension could redirect this to a workspace path by setting env N8N\_USER\_FOLDER or similar to the project (n8n allows some customization of its data folder). Regardless, n8n workflows can always be exported to JSON. The extension could automate exports to files in the workspace for version control, but that would be an extra step. The n8n-as-code extension essentially does this in reverse: it pulls JSON and lets you edit/save it as files, then pushes back to n8n. That extension requires an API key and host URL to connect to a running n8n – meaning it assumes you have an n8n server somewhere (local or remote). In our scenario, we’d be running that server invisibly inside VS Code. We could combine approaches: run n8n headless, and use a custom VS Code UI (or minimal WebView) to manipulate workflows (either via JSON editing or maybe a simplified custom canvas).  
* Integration Challenges: There are a few big ones: (1) Authentication/UI – as discussed, dealing with n8n’s login flow is the toughest problem for full UI embedding. You might try to start n8n in a special mode. One possibility: If you start n8n with environment variable N8N\_USER\_MANAGEMENT\_DISABLED=true *and* no owners exist yet, it might allow proceeding without login (some community hints suggest if no owner is created, it’s open, but this may not be officially supported). Alternatively, one could programmatically create a user via CLI or API and then simulate a login session in the WebView (e.g., inject the authentication token into localStorage or a cookie before loading the UI). This is complex and brittle. (2) License – n8n’s license could deter distribution as part of an OSS extension. If the extension is just for personal or internal use, it’s fine. But if publishing on the Marketplace or GitHub, one has to ensure compliance with the Sustainable Use License terms (for example, one condition is not offering the n8n editor as a service to users beyond a certain limit). In essence, bundling n8n in a widely distributed extension might raise legal questions, since it’s not a straightforward OSS license. (3) Resource usage – n8n is heavier than the others. It includes lots of nodes/integrations. Running it will take more memory/CPU. This isn’t a show-stopper, but something to consider for the user experience. (4) No existing full integration example – meaning you’d be writing the blueprint yourself. However, the *partial* integrations (like vscode-n8n-utils which helps launch n8n, or n8n-as-code which syncs JSON) provide some guidance. Those indicate that many in the community run n8n alongside VS Code rather than inside it.

Summary for n8n: While you *can* run n8n locally and even communicate with it from VS Code, embedding the complete n8n canvas UI is the most challenging of the four. The requirement of a running backend is common to all these tools, but n8n adds an authentication layer that isn’t easily suppressible. Also, from a product standpoint, n8n (being a standalone automation tool with its own desktop app) wasn’t designed with VS Code integration in mind, whereas Node-RED, for example, already has an extension. If seamless, local-first integration is the priority, n8n will likely involve more engineering (or perhaps isn’t the best fit) unless you’re okay with a non-visual or semi-visual approach (like editing JSON with a preview). The license is another strike against easy adoption in an OSS project. In short, n8n is powerful but would be a “heavy” integration both technically and legally.

## **Node-RED (Flow-Based IoT/Automation Tool)**

Node-RED is a well-known low-code tool originally from the IoT domain, now general-purpose, used to wire together hardware devices, APIs, and software services. It’s mature (a decade old) and lightweight (pure Node.js, typically running on Raspberry Pi and similar). Licensing is Apache 2.0, completely open and friendly to embed. Node-RED already has a history of being embedded in other contexts, and importantly for our use case, there is an existing VS Code extension (Node-RED for VS Code) that demonstrates successful integration.

* UI Embeddability: Node-RED’s editor is a single-page web app served from the Node-RED runtime (usually at http://localhost:1880). The VS Code extension by @formulahendry takes the approach of launching Node-RED and then opening a WebView that shows the editor UI. It provides commands like “Node-RED: Open” which effectively present the Node-RED canvas inside VS Code. Under the hood, the extension either iframes the URL or uses a WebView that loads the Node-RED HTML/JS/CSS. According to the extension’s documentation, you can even connect it to a remote Node-RED by setting the URL (for example, to use Node-RED running on a Raspberry Pi). By default though, it spawns a local instance. The fact this exists and has \~79k installs indicates the approach works well. One known issue was that VS Code’s older WebView did not support localStorage, and Node-RED’s editor uses localStorage for state. As a result, switching away from the WebView and back caused the Node-RED editor to reload (losing any unsaved edits). This was noted as a limitation and was expected to be resolved once VS Code allowed preserving localStorage in WebViews. This is a minor UI inconvenience rather than a blocker. Recent versions of VS Code *do* support state persistence in webviews (via retainContextWhenHidden), which likely mitigates this now. Other than that, Node-RED’s UI fits nicely. The extension basically proves that embedding Node-RED is already solved: when you trigger the command, the full Node-RED editor appears in a panel, and you can use it normally (drag nodes, deploy flows, etc.). The integration is so seamless that you can forget you’re inside VS Code, aside from the fact it’s in a panel.  
* Execution Engine Integration: Node-RED runs as a Node.js application (it’s essentially an Express app plus a runtime engine for flows). The extension includes Node-RED as an npm dependency and starts it up programmatically. When Node-RED starts, it will listen on a random local port (the extension settable default is 8008, or 127.0.0.2:8008 in their example). Once running, the WebView loads the editor from that local server. Node-RED’s engine executes the flows in the same process. For a VS Code extension, this is straightforward to manage: spawn the process or even require Node-RED as a module and call its start function with a custom settings configuration. The existing extension allows a custom settings.js to be specified via VS Code settings. This settings.js can define many things about Node-RED’s behavior (e.g., flow file location, enabling projects, etc.). So the extension developer has fine control. In short, Node-RED’s engine can absolutely run within the extension context, and because it’s just Node, it doesn’t introduce heavyweight dependencies beyond what VS Code already uses. It’s worth noting that Node-RED’s startup is quick and its memory usage is typically modest unless you load it up with many nodes.  
* Local Execution: Node-RED was built for local/offline scenarios (think IoT gateways with no internet). It requires no cloud services; everything runs on the local machine or local network. If your flows include HTTP requests or online APIs, then internet is needed for those nodes, but the platform itself is fully offline-capable. In an offline VS Code extension scenario, Node-RED shines: you can do file manipulations, hardware control, data processing, etc., without any external dependencies. Node-RED also has a huge library of community nodes (for every imaginable service or device). Those nodes are installed via npm. In the VS Code context, you would install nodes into Node-RED’s user directory. The extension could allow this (perhaps exposing a way to run npm install for Node-RED nodes). The existing extension might not cover an interface for installing additional nodes, but advanced users could drop them in the right folder. Regardless, offline usage is first-class.  
* Persistence & Workspace: By default, Node-RED stores its flows in a JSON file (e.g., flows\_\<hostname\>.json) in the user’s Node-RED directory. However, Node-RED supports a “projects” mode where the flow is tied to a git project. When projects are enabled, Node-RED will actually use the workspace folder (a local git repo) to store flows, along with a package.json and other metadata for the project. The VS Code extension’s documentation even shows how to enable projects via the editorTheme.projects.enabled setting in the Node-RED config. This means you can have your flows saved as files within the VS Code workspace and even version control them naturally. This feature is unique to Node-RED among the tools considered – it was explicitly designed to integrate with development workflows and Git. In practice, you could create a git repo for your automation flows, open it in VS Code, and with Node-RED’s project mode, any changes you make in the Node-RED canvas are saved to that repo (and you can commit them). This is a big advantage for a “local-first” philosophy. So Node-RED can achieve full persistence in the workspace with minimal effort (just a config flag). The other tools would require manual export/import for similar behavior.  
* Limitations & Notes: Node-RED’s integration is already in a usable state via the community extension. The known limitation was the editor reload on tab switch (due to localStorage), which is expected to be resolved in newer VS Code. Another small consideration: the extension spawns Node-RED on a specific address (they chose 127.0.0.2 to avoid conflicts on 127.0.0.1). This might have been to avoid clashing with any existing Node-RED or to bypass some CSP rules by using a slightly different host. In any case, as an integrator you’d ensure the host:port you use is allowed. The extension also exposes a setting for the URL, meaning users can connect to external Node-RED instances if they want. This flexibility is nice but not essential to our core goal (which is running local). No authentication issues: Node-RED by default doesn’t require login for local access (you can add optional admin auth, but typically it’s open on localhost). So you won’t face login prompts in the WebView. Licensing is Apache 2.0 – no problems merging with any extension license or modifying the code. Community & Support: Node-RED has an active community, and given an extension already exists, one could either use it as-is or fork/extend it. The current extension is somewhat basic (it opens Node-RED, that’s it), but it demonstrates everything working. You might enhance it to, say, sync flow files or interact with VS Code commands. At the very least, you know it’s possible and relatively simple.

Summary for Node-RED: Node-RED is the easiest and most battle-tested integration of the four. It was practically made to be embedded in local environments. The VS Code extension by Jun Han confirms that you can run the Node-RED UI in a WebView and control a local Node-RED instance. For a local-first, offline, open-source extension, Node-RED is a top choice, especially if your use case extends beyond just LLMs/AI into general automation. It has no serious integration blockers; any minor issues (like the storage persistence) have established solutions.

| ----- | ----- | ----- | ----- | ----- |

## **Recommendation**

Considering the above, the best suited tools for a seamless, local-first VS Code extension integration are Node-RED and Flowise. They offer the easiest paths and fewest obstacles:

* Node-RED is the most integration-ready: it’s lightweight, fully offline, and an official VS Code extension already demonstrates its viability inside a WebView. For general workflow orchestration (especially if your scenarios go beyond just LLMs/AI), Node-RED will integrate with minimal effort. It’s also very stable and has an Apache 2.0 license, meaning you can bundle/modify it freely. If your extension aims to provide a broad visual automation canvas, Node-RED is a safe bet. You will benefit from existing patterns (perhaps even reuse the open-source extension code) and get features like project-based flow files for free.  
* Flowise is a strong choice if your focus is specifically on AI agents and LLM workflows. It has a modern interface and is designed for the kinds of prompts/chains that AI developers work with. Technically, embedding Flowise should be only slightly more work than Node-RED: spawn a Node process and load a web UI. The absence of existing VS Code integrations is made up for by its clear separation of front/backend and permissive license. You won’t have to fight any authentication prompts or proprietary limitations. You may need to put in some effort to handle the database path and possibly provide an export mechanism so users can keep their flows in the workspace if desired. But these are achievable. Thus, for an AI-centric, offline VS Code extension (think “local LangChain canvas in VS Code”), Flowise is an excellent fit.

NodeTool, while appealing for local AI workflows, is less suited for easy embedding. Its reliance on a Python backend and AGPL license make it cumbersome for a VS Code extension. If one absolutely needs NodeTool’s advanced features (like running Stable Diffusion or video generation nodes inside VS Code), it’s possible, but expect to invest in creating a robust environment setup for users (possibly shipping a Python environment or at least detecting and guiding installation). Unless those specific capabilities are a must-have, Flowise could cover a lot of AI workflow needs with far less friction. n8n is not recommended for a seamless VS Code-embedded experience. It’s great as a standalone tool, but the combination of a non-webview-friendly authentication flow and a restrictive license makes it the outlier here. If one only wanted to trigger n8n workflows from VS Code or edit them as code, there are ways to do that (as evidenced by community extensions), but providing the *full* visual builder inside VS Code, offline, would be an uphill battle. For an open-source project aimed at local use, Node-RED covers much of the same general automation space without these headaches. In conclusion, Node-RED (for general workflows) and Flowise (for AI/LLM workflows) emerge as the top candidates for integration. They are both local-first and have permissive licensing, and their UIs can be embedded with manageable effort. A possible strategy is to take inspiration from the Node-RED VS Code extension for the WebView integration pattern and apply it to Flowise’s use case. This yields an extension where users can visually wire up logic (be it IoT devices or AI prompts) and execute everything on their local machine, all within the comfort of VS Code. Such an approach aligns perfectly with a local-first, offline-capable philosophy.
